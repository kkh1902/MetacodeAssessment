# AI Engineer 6주차 채점 기준서

> **강사: 변선영 / 주제: LLM 디코딩·프롬프트 엔지니어링·평가 지표·DSPy·Self-Correction**
> **채점 원칙: 문제당 0 or 10, 부분점수 없음 · 총 10문제 × 10점 = 100점**

## 채점 방식
- 각 문제별 배점: **10점** (O = 10점 / X = 0점, 이진 채점)
- O 판정: 문제지의 핵심 요구사항 **모두** 충족
- X 판정: 하나라도 미흡 / 미제출 / 실행 불가
- 감점 사유는 **20자 내외 핵심 키워드**로 기재 (문제 재진술·긴 부연 금지)

## Q1. 데이터 추출 에이전트의 디코딩 최적화

**요구사항 (문제지 기준)**
- `get_extraction_config()`가 결정론적 디코딩을 위한 파라미터 dict를 반환할 것
- 3개 키 모두 설정: `temperature`, `top_p`, `presence_penalty`

### O (10점)
- `temperature == 0.0` (무작위성 제거)
- `top_p == 1.0` (temperature=0과 결합 시 실질 greedy, 유효 범위 내)
- `presence_penalty == 0.0` (일관성 유지)

### X (0점)
- `top_p == 0.0` 또는 유효 범위 밖 값 제출 (API 호출 실패 유발)
- 키 누락 / 결정론적 의도 불명
- 감점 사유 예: `top_p 0.0 (유효범위 밖)`, `temperature > 0`

## Q2. Few-shot을 활용한 구조적 데이터 추출

**요구사항**
- `FEW_SHOT_EXAMPLES`에 user/assistant 턴 쌍으로 예시 작성
- `messages = [system] + few_shot + [user]` 순서로 구성
- 출력 포맷: `[{"person": ..., "org": ...}]` JSON 배열

### O (10점)
- Few-shot 예시 **최소 1쌍 이상** (user 입력 + assistant JSON 응답)
- System 프롬프트에 "JSON 배열만 출력, 설명 금지" 취지 명시
- `messages.extend(FEW_SHOT_EXAMPLES)` + 실제 `news_text`를 user로 append

### X (0점)
- Few-shot 예시 없음 (zero-shot 상태)
- System 프롬프트 미작성 또는 포맷 규칙 부재
- assistant 예시 응답이 JSON 배열 형식 아님
- 감점 사유 예: `Few-shot 예시 없음`, `JSON 배열 포맷 미지정`

## Q3. 구조화된 데이터 추출 + Self-Correction 프롬프트

**요구사항**
- `system_prompt`에 역할(persona)과 JSON 출력 규칙 명시
- `error_message`가 있을 때 재시도용 user_content가 에러를 반영해 수정 유도
- 필수 필드(`store_name`, `date`, `items`, `total_amount`) 스키마 안내

### O (10점)
- System 프롬프트에 "순수 JSON만, 설명·마크다운 금지" 취지 포함
- 필수 필드 및 타입(`total_amount: int`, `date: YYYY-MM-DD`) 명시
- `error_message` 분기로 재시도 시 에러 내용을 프롬프트에 주입

### X (0점)
- 시스템 프롬프트 공란 또는 포맷 규칙 부재
- error_message 분기 없음 (항상 동일 프롬프트)
- 스키마 필드 미안내
- 감점 사유 예: `error_message 분기 없음`, `스키마 미명시`

## Q4. 프롬프트 엔지니어링 e2e (감정 분석 JSON)

## Q5. Chain-of-Thought 추론 (재고 퍼즐)

**요구사항**
- `standard_prompt`: Zero-shot (가이드 없이 바로 답 요청)
- `cot_prompt`: 단계별 사고 유도 문구 포함
- CoT 실행 결과에 "112" 포함해야 자동 채점 통과

### O (10점)
- standard와 cot 두 프롬프트 모두 작성
- CoT 프롬프트에 "step by step" / "단계별" / 단계 템플릿 중 하나 이상 포함
- CoT 실행 결과에 정답 `112` 포함 (자동 채점 `verify_logic` 통과)

### X (0점)
- 두 프롬프트 중 하나 이상 공란
- CoT에 추론 유도 문구 전혀 없음 (standard와 동일)
- 정답 112 미포함
- 감점 사유 예: `cot_prompt 공란`, `정답 112 미포함`

## Q6. ROUGE / BLEU 지표

**요구사항**
- `rouge_results['_______']` 빈칸 채우기 → `rougeL_fmeasure` (또는 `rougeL_fmeasure` 등가 키)
- 서술형 2문항 답변

### O (10점)
- 빈칸 `rougeL_fmeasure` (또는 `rougeLsum_fmeasure` 등 LCS F1 계열) 기재
- 서술형 4번: 동의어/패러프레이즈에 n-gram 지표가 취약해 점수가 하락한다는 취지
- 서술형 5번: **BERTScore** (또는 BLEURT/BARTScore 등 의미론적 임베딩 기반) 언급

### X (0점)
- `rougeL_precision` / `rougeL_recall` 등 F1이 아닌 값 또는 잘못된 키
- 서술형 미작성 또는 핵심 키워드 미언급
- 감점 사유 예: `rougeL_fmeasure 아님`, `BERTScore 미언급`

## Q7. 통계적 vs 의미론적 평가 지표 비교

**요구사항**
- `rouge_scorer.RougeScorer(['rougeL']).score(ref, pred)`로 ROUGE-L F1 추출
- `bert_score.score([pred], [ref], lang='ko')`로 BERTScore F1 추출
- `return rouge_l_score, bert_f1_score`
- 제공된 assert 검증 통과 (`rouge_val < 0.2`, `bert_val > 0.8`, `bert_val > rouge_val`)

### O (10점)
- 두 지표 모두 구현, `lang='ko'` 또는 다국어 모델 명시
- `.fmeasure` / `.item()` 스칼라 반환
- assert 전부 통과

### X (0점)
- 한쪽 지표 누락
- BERTScore `lang` 미지정 또는 잘못된 언어 (`en` 고정 등)
- assert 실패
- 감점 사유 예: `BERTScore lang 미지정`, `assert 실패`

## Q8. PyTorch Contrastive Decoding 구현

**요구사항 (문제지 명시)**
1. Adaptive Masking: Expert 로짓 상위권 외 필터링
2. Contrastive Logits: `expert - τ * amateur`
3. Softmax 및 argmax 선택
4. `torch`와 `torch.nn.functional`만 사용 (외부 라이브러리 금지)

### O (10점)
- 위 4가지 단계 **모두** 구현 (Masking 방식은 top-k, plausibility constraint `p >= α·max_p` 모두 인정)
- 반환값이 `int` (argmax 인덱스)
- 제공된 테스트의 기대값(`0`)을 반환하면 완전 통과이나, 구현이 문제지 4요건을 충족하고 로직이 일관되면 O 인정

### X (0점)
- 4단계 중 누락 (예: Masking 없음, 감산 수식 틀림)
- `sklearn`/`numpy` 등 외부 라이브러리 사용
- 함수가 `pass`로 비어있음
- 감점 사유 예: `Adaptive Masking 미구현`, `감산 수식 오류`

## Q9. DSPy 설계

**요구사항**
- **9-1**: `GuardrailSignature`에 `user_input = InputField(desc=...)`, `is_safe = OutputField(desc=...)` 정의
- **9-2**: `GuardrailAgent`에서 `self.analyze_safety = dspy.ChainOfThought(GuardrailSignature)`, `forward`에서 호출
- **9-3**: decoding config에 `top_p` (nucleus, 0.95 수준), `repetition_penalty` (>1.0, 예: 1.2)

### O (10점)
- 3개 서브문항 **모두** 충족
- `InputField`/`OutputField`에 `desc` 인자 실제로 기재
- `ChainOfThought` 래퍼 사용
- `top_p=0.95`(또는 0.9~0.95 범위), `repetition_penalty >= 1.1`

### X (0점)
- 서브문항 중 하나라도 공란/오류
- `desc` 미기재 / 필드 타입 잘못 사용 (`dspy.Predict` 등)
- `top_p` 또는 `repetition_penalty` 키 누락
- 감점 사유 예: `ChainOfThought 미사용`, `repetition_penalty 누락`

## Q10. Self-Correction 프롬프트 개선 루프

**요구사항**
- `critic_prompt`: 4가지 조건(글자수 120-150, 필수단어 3종, 금지단어 3종, 해시태그 맨 끝 3개)을 항목별로 평가하도록 유도
- `improvement_instruction`: 피드백을 반영해 "원래 지시사항"을 강화된 프롬프트로 재작성 지시
- 최종 `final_text`가 자동 채점(`evaluate_final_output`)에서 **60점 이상** 달성

### O (10점)
- critic_prompt에 4가지 조건 각각 검증 지시 포함
- improvement_instruction이 피드백(feedback)을 입력으로 받아 개선된 프롬프트를 출력하도록 설계
- 자동 채점 결과 **60/100점 이상** (최소 2개 조건 통과) — 루프 수렴성 확인

### X (0점)
- critic_prompt 또는 improvement_instruction 공란 / 제공된 f-string 그대로 둠
- 4가지 조건 중 명시 검증 항목 누락
- 자동 채점 결과 < 60점 (루프가 목표에 수렴하지 못함)
- 감점 사유 예: `critic_prompt 공란`, `자동 채점 < 60점`

## 공통 주의사항

1. **파일명이 다르거나 접근이 달라도** 문제지 요구사항을 모두 충족하면 O
2. **정답지와 완전 일치하지 않아도** 구현 방식이 다른 유효 해법이면 O (예: Q8의 masking 방식)
3. **수강생 제출 원본 파일은 절대 수정/삭제하지 말 것**
4. API 키 노출 등 민감정보는 코드에서 확인만 하고 감점 사유에는 반영하지 않음
5. 실행 결과 셀이 저장되어 있으면 재실행 없이 그것으로 채점 가능
