- **총점**: 100점 (Q1~Q10, 각 10점)
- **채점 원칙**: **문제당 0점 또는 10점** (부분점수 없음, 문제지 명시)
  - 문제지 규정: "출력 결과 일치"가 원칙, "코드의 다양성 인정"도 명시
  - 동일한 요구사항을 충족하고 출력 구조/수치가 합리적이면 O
  - 하나라도 핵심 요구사항 누락 → 0점

## Q1. 멀티 모델 임베딩 비교 벤치마크 (10점)

**문제지 요구사항**
- `EmbeddingBenchmark` 클래스의 5개 메서드 구현
- 로컬: `paraphrase-multilingual-MiniLM-L12-v2`
- API: `text-embedding-3-small` (OpenAI)
- 코사인 유사도 계산
- `evaluate` 반환 dict 키: `accuracy`, `precision`, `recall`, `avg_sim_positive`, `avg_sim_negative`, `elapsed_ms`
- threshold 기준 이진 분류 후 metric 계산

**O 판정 기준**
- 두 임베딩 함수 모두 numpy array 반환
- 코사인 유사도 공식 정확 (`dot / (norm * norm)`)
- accuracy/precision/recall 계산 공식 정확
- pos/neg 평균 유사도 각각 계산
- 최종 출력표가 로컬/API 두 컬럼으로 생성

**X 판정 사례**
- 모델명 오인 (한국어 모델이 아닌 영어 전용 모델 사용)
- accuracy 식을 (correct/total)가 아닌 다른 식으로 계산
- `elapsed_ms`에 초 단위 값을 그대로 기록

## Q2. 임베딩 차원 축소 + 시각화 분석 (10점)

**문제지 요구사항**
- SentenceTransformer로 임베딩 생성
- PCA(n_components=2)로 2차원 축소
- 실루엣 스코어로 클러스터 품질 평가
- 카테고리별 2D 중심점 계산
- 카테고리 간 중심점 거리 출력

**O 판정 기준**
- PCA 정상 수행 (2차원 결과)
- `silhouette_score` 호출 (원본 임베딩 또는 PCA 축소 결과 둘 다 허용 — 문제지 문구 해석 자유)
- 카테고리별 평균 좌표로 centroid 계산
- `l1 < l2` 조합별 거리 출력 (중복 쌍 방지)

**X 판정 사례**
- PCA 미사용 또는 차원이 2가 아님
- silhouette 계산 생략
- centroid 계산 오류 (평균 아님)
- 중복 쌍 출력 (ex: tech-food, food-tech 둘 다)

## Q3. ChromaDB CRUD + 메타데이터 필터링 (10점)

**문제지 요구사항**
- in-memory ChromaDB 클라이언트
- 컬렉션 (cosine distance) 생성
- 12개 문서를 임베딩 + 메타데이터와 함께 적재
- 3가지 검색 수행:
  1. 기본 시맨틱 검색 (top 3)
  2. `category == kubernetes AND difficulty != beginner`
  3. `year >= 2024 AND difficulty IN [beginner, intermediate]`
- `doc_003` difficulty를 `advanced`로 업데이트
- `doc_009` 삭제

**O 판정 기준**
- `metadata={"hnsw:space": "cosine"}` 지정
- 모든 add 4개 파라미터 포함 (ids, documents, embeddings, metadatas)
- `$and`, `$ne`, `$gte`, `$in` 연산자 사용
- update/delete 정상 수행 → 최종 카운트 11

**X 판정 사례**
- cosine space 미지정
- where 필터를 Python 후처리로 구현
- update/delete 누락

## Q4. ETL 파이프라인 (10점)

**문제지 요구사항**
- `sliding_window_chunk(text, chunk_size=150, overlap=30)`: 문자 단위 슬라이딩 윈도우
- `extract_keywords(text, top_n=5)`: 2글자 이상 단어 TF 상위 N개
- `build_chunk_metadata`: `title`, `source`, `chunk_index`, `char_count`, `keywords` (쉼표 `,` join, 공백 없음)
- `run_etl_pipeline`: Extract → 청킹 → 메타데이터 → 임베딩 → ChromaDB add

**O 판정 기준**
- 슬라이딩 윈도우 step = `chunk_size - overlap`
- 키워드 빈도 기반 top_n 추출
- `keywords` 포맷: `"kw1,kw2,kw3,..."` (쉼표 뒤 공백 없음)
- 컬렉션에 전체 청크가 metadata 포함하여 적재

**X 판정 사례**
- 청크 step 오류 (overlap 미적용)
- `keywords`를 list로 저장 (ChromaDB 메타데이터는 scalar만 허용)
- 쉼표 뒤 공백 포함
- 키워드 추출에 빈도 아닌 다른 방식 사용

## Q5. BM25 + Vector 하이브리드 검색 (10점)

**문제지 요구사항**
- BM25Okapi 인덱스 생성 (공백 토큰화)
- ChromaDB 벡터 검색
- RRF 융합: `Σ 1/(k + rank)` (k=60)
- `hybrid_search(query, top_k, mode)`: "bm25" / "vector" / "hybrid" 지원
- hybrid 모드는 각 검색에서 `top_k * 2` 가져온 후 RRF로 상위 `top_k` 반환

**O 판정 기준**
- BM25와 벡터 검색 각각 `(index, score, doc)` 튜플 리스트 반환
- RRF 공식이 정확 (rank 1-indexed)
- mode 분기 정상 작동

**X 판정 사례**
- RRF에서 rank를 0-indexed로 쓰거나 `1/rank`만 사용
- score 부호 반대 (distance를 그대로 score로 사용)
- `top_k * 2` 미적용

## Q6. Hit Rate@K / MRR@K (10점)

**문제지 요구사항**
- `hit_rate_at_k`: top-K 안에 관련 문서 하나라도 있으면 1, 아니면 0
- `mrr_at_k`: 첫 번째 관련 문서의 역순위 (rank는 1부터)
- `evaluate_search_system`: ks=[1, 3, 5]별 평균 반환
- bm25/vector/hybrid 각각 평가 후 표 출력

**O 판정 기준**
- hit_rate / mrr 공식 정확
- ks별로 독립 루프
- 최종 표에 9개 행 (3 modes × 3 ks)

**X 판정 사례**
- mrr에서 rank를 0부터 시작
- 관련 문서 없는 경우 1이 아니라 에러 발생
- 모든 k에서 전체 결과 재사용 없이 혼동

## Q7. FastAPI 벡터 검색 API 서버 (10점)

**문제지 요구사항**
- Pydantic 모델 3개: `SearchRequest`, `SearchResult`, `SearchResponse`
- `InMemoryCache`: ttl 기반 만료, `make_key` 해시, `hit_count` property
- `RateLimiter`: sliding window, max_requests=30, window=60
- `rate_limit_middleware`: 초과 시 429
- `/search` 엔드포인트: 캐시 확인 → hybrid_search → 캐시 저장 → 응답
- `/health` 엔드포인트

**O 판정 기준**
- 3개 Pydantic 모델 필드 모두 정의
- 캐시 TTL 만료 로직 정상
- Rate limiter sliding window 구현 (오래된 요청 필터링)
- 서버가 백그라운드 스레드에서 실행되고 httpx 테스트 통과

**X 판정 사례**
- Pydantic 필드 누락 또는 타입 오류
- 캐시 TTL 미적용 (영구 저장)
- RateLimiter에서 `time.time()` 비교 오류
- 서버 스레드 미실행으로 httpx 호출 실패

## Q8. 자동 청킹 전략 최적화 (10점)

**문제지 요구사항**
- `chunk_fixed_size`: 고정 크기 + overlap
- `chunk_by_sentence`: 마침표 기준 분리 후 2~3문장씩 그룹
- `chunk_semantic`: 연속 문장 유사도 < threshold에서 분할
- `evaluate_chunking_strategy`: 쿼리별 top-3 결과에 answer_keyword 포함 여부로 Hit Rate@3
- 3개 전략 비교 후 최적 전략 자동 선택

**O 판정 기준**
- 3개 청킹 함수가 모두 list[str] 반환
- semantic chunking이 임베딩 + 유사도 기반으로 실제 분할 수행
- Hit Rate@3이 각 전략별로 개별 계산
- `best`가 최고 점수 전략명으로 설정

**X 판정 사례**
- semantic chunking에서 임베딩 미사용
- 전략 간 comparison 로직 누락
- best 하드코딩

## Q9. 비동기 배치 임베딩 성능 비교 (10점)

**문제지 요구사항**
- `_split_batches`: batch_size 단위 분할
- `process_sync`: 순차 처리 + 재시도 + 진행률
- `_embed_batch_async`: `run_in_executor`로 비동기 실행
- `process_async`: `asyncio.gather`로 동시 실행
- 동기/비동기 결과 비교 (속도 + 유사도)

**O 판정 기준**
- 배치 분할 정확
- 재시도 루프 (max_retries + 1회)
- 실패 시 `failed_batches`에 인덱스 기록
- `asyncio.gather` 사용
- 결과 배열 shape 일치 (sync/async)

**X 판정 사례**
- `asyncio.gather` 대신 순차 await (비동기 이점 없음)
- 재시도 로직 누락
- `run_in_executor` 미사용
- vstack 누락으로 shape 불일치

## Q10. RAG 평가 파이프라인 통합 (10점)

**문제지 요구사항**
- `RAGEvaluationPipeline` 클래스
- ChromaDB 문서 적재 (cosine)
- `retrieve`: top-k 문서 + 인덱스 반환
- `generate_answer`: LLM으로 답변 생성 (참고 문서만 사용)
- `evaluate_retrieval`: MRR@K
- `evaluate_answer_rouge`: ROUGE-L F1 (LCS 기반)
- `evaluate_faithfulness`: LLM으로 JSON 판정
- `run_full_evaluation`: 통합 리포트 출력

**O 판정 기준**
- 모든 메서드 구현
- ROUGE-L은 LCS DP 기반 F1 (token 단위는 단어/문자 둘 다 허용 — 표준 ROUGE 변형)
- Faithfulness JSON 파싱 (`faithful`, `score`, `reason`)
- 최종 리포트에 MRR, 평균 ROUGE-L, faithful 비율, 평균 score 모두 출력

**X 판정 사례**
- ROUGE-L을 단순 n-gram overlap으로 계산 (LCS 아님)
- Faithfulness JSON 대신 자유 텍스트 응답
- LLM 호출 미구현 (답변이 None)
- 종합 리포트 누락
