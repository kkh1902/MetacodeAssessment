# 13주차 채점기준서 — 정채진 강사님

> **채점 원칙: 각 서브문제 0 or 만점, 부분점수 없음** (문제지 명시 기준)
> 총점 100점 = 9문제 × 25개 서브문제 (배점 상이)

## 채점 방식
- 서브문제별 0점(미충족) 또는 만점(문제지 요구사항 모두 충족)
- 서술형: 문제지가 요구하는 핵심 키워드/항목 수를 충족해야 만점
- **서브문제별 0점 조건은 문제지가 각 문제 말미에 명시** — 본 기준서는 그 규칙을 그대로 차용

## 공통 규칙 (문제지 명시)
- **모든 스크린샷/터미널 캡처에 `date && hostname` 출력 포함 필수**
  - 출력 없는 캡처는 본인 작업 증빙으로 인정하지 않음 → 해당 서브문제 0점
- 제출 형식: Google Docs / Notion 링크 (PDF 허용)
- 코드 파일(.yml/.json/.tf/.py/.conf)은 코드 블록으로 문서에 포함
- 다이어그램은 `.excalidraw` 또는 `.drawio` 소스 파일 함께 제출

## 주차 주제
Agent Service Deployment Basic — Dockerfile/compose, Nginx 리버스 프록시, AWS ECS/Fargate, GCP Cloud Run, Terraform IaC, GitHub Actions CI/CD, 멀티클라우드 아키텍처, 프로덕션 운영(OOM/503/배포 전략), 구조화 로깅·비용 최적화.

## 문제 1: Docker 기초 및 Dockerfile 작성 (12점)

### 1-1. Backend Dockerfile 직접 작성 (5점)
**요구사항** (문제지 verbatim):
- 베이스: `python:3.11-slim`
- healthcheck용 `curl` 설치 (apt 캐시 정리 포함)
- `WORKDIR /app`

- 의존성 파일: `pyproject.toml`, `uv.lock`
- 포트: 8000

- **제출물**: 완성된 Dockerfile, `docker build` 성공 스크린샷, **서술(1.5점)**: 의존성 파일 선복사 이유를 본인 Dockerfile의 라인 번호를 참조하여 설명

**X (문제지 0점 조건)**:
- 빌드 실패하는 Dockerfile
- 스크린샷 없음
- 서술에서 본인 Dockerfile 라인 번호 미참조 (일반론만 서술)

### 1-2. 고장난 docker-compose.yml 수정 (4점)

- 예상 오류: version(deprecated 논외)/env_file 경로/healthcheck 포트(3000→8000)/frontend의 networks(agent-network 정의됨 vs backend의 app-network 불일치)/depends_on 조건 등
**O**: 수정된 docker-compose.yml + 각 오류 무엇/왜/어떻게 설명 + 성공 캡처
**X (문제지 0점 조건)**:
- 3개 미만 수정
- 서비스가 정상 실행되지 않음
- 스크린샷 없음

### 1-3. 환경변수 보안 사고 대응 (3점)

- **(a)** 3단계 긴급 대응 절차 (1.5점) — 1순위 API 키 무효화 필수

## 문제 2: Nginx 리버스 프록시 및 멀티 컨테이너 (12점)

### 2-1. nginx.conf 직접 작성 (5점)
**요구사항**:
- `events`, `http`, `server` 3개 블록
- 정적 파일 서빙: `location /` — root `/usr/share/nginx/html`, SPA `try_files`
- 백엔드 API 프록시: `location /api/` — `proxy_pass`가 `/api/` prefix 제거하여 backend에 전달 (`GET /api/ask?q=hello` → `GET http://backend:8000/ask?q=hello`)
- SSE 3종 세트: `proxy_buffering off`, `proxy_cache off`, `chunked_transfer_encoding on`
- `proxy_http_version 1.1`
- `location /health` — `return 200 "OK\n"`
- 제출물: nginx.conf + `/api/` 경로 요청이 backend로 전달되는 결과 스크린샷
**O**: 위 모든 요소 포함 + `/api/` 프록시 동작 캡처
**X (문제지 0점 조건)**:
- SSE 설정 3종 세트 누락
- `proxy_pass`에서 `/api/` prefix가 제거되지 않음
- 스크린샷 없음

### 2-2. 컨테이너 네트워크 디버깅 (4점)
**요구사항**:
- **(a)** 502 Bad Gateway 원인 3가지 (확률 높은 순) (1.5점)
- **(b)** 각 원인별 프로젝트 특화 디버깅 명령어 (1.5점) — `agent-network` 등 실제 프로젝트명 사용 필수
- **(c)** Compose 브릿지 네트워킹에서 서비스명 통신 원리 (1점) — Docker 내장 DNS(127.0.0.11) 포함
**O**: 3항목 모두 충족 + 네트워크/DNS 원인 언급 + 프로젝트 네트워크명 명시
**X (문제지 0점 조건)**:
- 네트워크/DNS 해석을 원인으로 언급하지 않음
- 디버깅 명령어에 `agent-network` 등 프로젝트 특정 이름을 사용하지 않음

### 2-3. 헬스체크 설정 작성 (3점)
**요구사항**:
- **(a)** backend healthcheck YAML (0.5점) — `test`, `interval`, `timeout`, `retries`, `start_period`
- **(b)** 각 파라미터 역할 설명 (1점)
- **(c)** 시나리오: `start_period=0`이고 backend 시작 30초 소요 시 이벤트를 시간순 서술 (1.5점) — frontend `depends_on: condition: service_healthy`에 미치는 영향
**O**: 세 항목 모두 + depends_on/service_healthy로 인한 frontend 시작 지연/실패 설명
**X (문제지 0점 조건)**:
- `depends_on` + `service_healthy` 조건에 의한 frontend 시작 지연/실패 미설명
