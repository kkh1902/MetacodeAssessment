### 문제 2: CSV → SQLite 재고 조회 (10점)
**요구사항**:
- **2-1** `init_inventory_db(csv_data) -> sqlite3.Connection`
  - 테이블 스키마: `item_id TEXT PRIMARY KEY, name TEXT, category TEXT, quantity INTEGER, location TEXT, status TEXT, last_updated TEXT`
  - `db.row_factory = sqlite3.Row`
  - `quantity`를 INTEGER로 변환하여 삽입
- **2-2** `search_inventory(db, query) -> list[dict]`
  - name OR category에서 키워드 매칭, `LOWER` 사용 (대소문자 무시)
  - `LIMIT 10`
  - dict 리스트 (JSON 직렬화 가능) 반환
- 검증: `'dell'`→2건 (기대 출력은 표기 오기 "1건"이나 실제 2건 출력), `'peripherals'`→4건, `'xyz_not_exist'`→0건
#### O (10점) — 모두 충족
- [ ] 테이블 스키마 일치 + `row_factory = Row`
- [ ] `LOWER` 사용한 대소문자 무시 검색
- [ ] `LIMIT 10`
- [ ] dict 리스트 반환 (JSON 직렬화 가능)
- [ ] 검증 3개 모두 기대 결과 (Dell 2건, Peripherals 4건, 없음 0건)
#### X (0점) 사례
- `LOWER` 미사용 (대소문자 매칭 실패)
- `LIMIT` 누락
- `row_factory` 미설정
- 미제출

- [ ] 검증 4개 출력과 맥락 일치

### 문제 5: Confirm Gate + 멱등성 (10점)
**요구사항**:
- **5-1** `create_ticket(...)`
  - `confirm=False` → `"[미리보기] 제목: {title} | 우선순위: {priority} — confirm=True로 생성하세요."` 반환, store 변경 없음
  - `confirm=True` → ticket_id `TKT-001` 형식 (`len(store)+1`), 딕셔너리 `{ticket_id, title, body, priority, status:"open", created_at: ISO UTC}` 추가/반환
- **5-2** `create_ticket_idempotent(...)`: 위 확장
  - 멱등성 키 `sha256((title+body).encode()).hexdigest()[:16]`
  - 동일 키 존재 시 기존 티켓 반환 (중복 생성 금지)
  - 새 티켓에 `idempotency_key` 필드 포함
- 검증 4 시나리오 출력 일치
#### O (10점) — 모두 충족
- [ ] 미리보기 문자열 포맷 정확
- [ ] ticket_id `TKT-001/002` 포맷 (`len+1`)
- [ ] sha256 해시 16자리 멱등성 키 사용
- [ ] 중복 요청 시 기존 티켓 반환 (store 길이 불변)
- [ ] `created_at` ISO UTC 포함
#### X (0점) 사례
- `confirm=False`에서 store에 추가
- 멱등성 키 미구현 (중복 생성됨)
- `idempotency_key` 필드 누락
- ISO UTC 미사용
- 미제출

### 문제 6: 입력 검증과 보안 (10점)
**요구사항**:
- **6-1** `sanitize_string(value, max_length=500) -> str`
  1. str 타입 아니면 ValueError
  2. strip
  3. `CONTROL_CHARS` 제거
  4. `max_length` 초과 시 자르기
  5. `DANGEROUS_PATTERNS` 매칭 시 ValueError
  6. 정제된 문자열 반환
- **6-2** `validate_doc_id(doc_id) -> str`
  - 화이트리스트 정규식 `^[a-z0-9]([a-z0-9\-]{0,48}[a-z0-9])?$` 사용 (1~50자, 하이픈 시작/끝 불가)
  - `..`, `/`, `\`, 널 바이트, 대문자, 빈 문자열 차단
  - 통과 시 strip된 doc_id 반환, 실패 시 ValueError
- 검증 출력 모두 일치
#### O (10점) — 모두 충족
- [ ] sanitize 6단계 처리 순서 준수
- [ ] `CONTROL_CHARS`/`DANGEROUS_PATTERNS` 사용
- [ ] validate_doc_id 정규식 통과/실패 모두 올바름
- [ ] SQL/XSS/NoSQL 패턴 차단 테스트 통과
- [ ] path traversal / 대문자 / 길이 초과 차단
#### X (0점) 사례
- 처리 순서 어긋남 (예: DANGEROUS_PATTERNS 매칭 후 무시)
- 정규식 부정확
- 검증 테스트에서 공격 통과
- 미제출
