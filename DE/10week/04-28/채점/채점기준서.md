# 10주차 채점 기준서

## 채점 방식

- 각 문제별 배점: **10점**
- O (10점): X-1, X-2 세부문제 **모두** 통과
- X (0점): X-1, X-2 중 하나라도 미충족
- 총점: **100점** (10문제)
- 채점 방식: 자동채점 60% + 구조화 수동채점 40%
- **자동채점 50% 미만 시 실격 처리**

## 공통 0점 조건 (모든 문제 적용)

아래 중 하나라도 해당하면 해당 문제 **전체 0점**:

- 캡처/로그/실행결과 증빙 없음
- 수정 전/후 비교 없음
- 가설 2개 미만 (가설 요구 문제)
- 최종 원인 명확히 특정 못함

- `.docx` / `.hwp` 등 문서 파일만 단독 제출
- 코드를 이미지(캡처/사진)로 제출

## 문제 5: 죽지 않는 OOM — k8s Spark Executor 메모리 진단 [10점]

**주제**: k8s 리소스 관리 | **난이도**: 심화

**상황**: 제공된 SparkApplication을 k3d 클러스터에 배포하면 executor pod이 OOMKilled를 반복. driver pod은 정상 Running. 버그 2개 존재. Bug 1(`memoryOverhead: "64m"`)이 OOMKilled의 직접 원인이며, Bug 2(`javaOptions` deprecated 플래그)는 설정 품질 문제.

> **버그 정리**
> - **Bug 1** (필수): `memoryOverhead: "64m"` → container limit = 512m+64m = 576m. Python worker가 700MB 할당 시도 → 576m 초과 → OOMKilled
> - **Bug 2** (필수): `javaOptions: "-XX:+PrintGCDetails -XX:+PrintGCDateStamps"` → JDK 9+ deprecated 플래그. Bug 2만 수정 시 OOMKilled 지속.

### O (10점) — 아래 항목 **모두** 충족

**문제 5-1: OOMKilled 원인 분석**

| # | 통과 조건 (코드/설명) | 통과 조건 (캡처/증빙) |
|---|---|---|
| 2-1a | kubectl 실측 데이터 수집 (describe + top + JVM settings) | OOMKilled 이벤트 + 실측 메모리 사용량 캡처 |
| 5-1b | Bug 1(memoryOverhead) 특정 + container limit 수치 계산 (memory+overhead=576m < 700MB) + Bug 2(javaOptions) 특정 | JVM heap 크기 vs container limits 비교 캡처 |

**문제 5-2: 설정 수정 및 executor 안정화 증명**

| # | 통과 조건 (코드/설명) | 통과 조건 (캡처/증빙) |
|---|---|---|
| 5-2a | Bug 1·2 모두 수정 + memoryOverhead 계산 근거 포함 (Python 700MB 수용 가능 수치) | executor Completed + top으로 limits 이내 확인 |
| 5-2b | SparkApplication Completed 상태 캡처 | OOMKilled 재발 없이 안정 완료 증명 |

### X (0점) 추가 조건
- 공통 0점 조건 해당
- 버그 1개만 수정한 경우 (2개 모두 수정 필요)

## 채점 시 주의사항

1. **수강생 제출 원본 파일은 절대 수정/삭제하지 말 것**
2. **파일명이 다르더라도** 동일한 기능을 구현했으면 해당 문제로 인정
3. **캡처에 `date && hostname` 없으면 무조건 0점** — 예외 없음
4. **중간 과정 캡처 필수** — 최종 성공 캡처만 제출 시 0점
5. **정답과 다른 접근이라도** 요구사항을 모두 충족하면 O
6. **코드만 제출, 캡처 없음** → 전 문제 0점 (증빙 원칙)
7. 문제 4-2b, 5-2b, 6-2b, 7-2b는 자동채점 항목 — 코드 실행 결과로만 판단
