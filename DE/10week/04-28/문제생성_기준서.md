# 10주차 문제 생성 기준서

## 목적

수강생이 **개인 노트북(RAM 8GB)** 환경에서 문제를 수행할 수 있도록,
인프라 리소스·버그 설계·기술 스택에 대한 기준을 정의한다.

---

## 1. 목표 스펙

| 항목 | 기준값 |
|---|---|
| 최소 RAM | 8GB |
| 권장 RAM | 16GB |
| 최소 CPU | 2코어 (4스레드) |
| OS | Windows 11 (WSL2) / macOS (Intel & M1/M2) / Ubuntu |
| Docker Desktop 메모리 설정 | **4GB** (기본값 2GB에서 반드시 변경) |
| Docker Desktop CPU 설정 | 2코어 이상 |

### 노트북 RAM 실사용 가능량 (8GB 기준)

| 용도 | 점유량 |
|---|---|
| OS (Windows/macOS) | ~2GB |
| 브라우저 + IDE | ~1.5GB |
| Docker Desktop 오버헤드 | ~500MB |
| **Docker 컨테이너에 쓸 수 있는 양** | **~4GB** |

> **전원 연결 필수** — 배터리 모드에서는 CPU 성능이 40~60% 수준으로 떨어져
> 타이밍 기반 버그가 의도치 않게 발현되거나 job이 타임아웃될 수 있다.

| 서비스 | mem_limit | 비고 |
|---|---|---|
| Zookeeper | 512m | TICK_TIME 조정 시 변동 없음 |
| Kafka broker | 768m | heap 포함 |
| Kafka UI | 512m | |
| Spark master | 1g | |
| Spark worker | 1536m | SPARK_WORKER_MEMORY=1g |
| PostgreSQL | 512m | |

| Airflow scheduler | 512m | |

### 2-2. 문제별 전체 예산 한도

| 조건 | 기준 |
|---|---|
| 전체 컨테이너 mem_limit 합산 | **3.5GB 이하** |
| 동시 기동 컨테이너 수 | **6개 이하** |
| Kafka 브로커 수 | **1개** (문제 의도상 다수 필요 시 최대 2개) |
| Spark worker 수 | **1개** |

## 3. Docker Compose 작성 규칙

### 3-1. 필수 항목

```yaml
# 모든 서비스에 mem_limit 명시
deploy:
  resources:
    limits:
      memory: 512m
```

### 3-2. Kafka 설정 기준

```yaml
# 브로커 수: 1개
KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 1

# Heap: 버그용 / 정상용 구분
KAFKA_HEAP_OPTS: "-Xms256m -Xmx256m"   # 버그용 (의도적으로 작게)
KAFKA_HEAP_OPTS: "-Xms512m -Xmx1g"     # 정상용

# ZK 타임아웃 버그 설계 시: tickTime을 500으로 낮춰야 실제 적용됨
# (ZK minSessionTimeout = 2 * tickTime, 기본 2000ms면 최소 4000ms로 협상됨)
ZOOKEEPER_TICK_TIME: 500
KAFKA_ZOOKEEPER_SESSION_TIMEOUT_MS: "3000"   # 버그용
KAFKA_ZOOKEEPER_SESSION_TIMEOUT_MS: "18000"  # 정상용
```

### 3-3. Spark 설정 기준

```yaml
# worker 수: 1개
# memory: 1g (노트북 기준 최대)
SPARK_WORKER_MEMORY: 1g
SPARK_WORKER_CORES: 1
```

### 3-4. 금지 사항

- `mem_limit` 미설정 금지 — 컨테이너가 호스트 RAM을 무제한 점유
- `replication_factor: 3` 브로커 1개 환경에서 사용 금지
- Spark worker 2개 이상 금지 (RAM 예산 초과)

## 4. 버그 설계 원칙

### 4-1. 기동 가능성

| 버그 유형 | 허용 여부 | 비고 |
|---|---|---|
| 기동 직후 crash (의도된 OOM 등) | ✅ | 문제 1처럼 crash 자체가 진단 대상인 경우 |
| 기동 자체 불가 (이미지 없음, 포트 충돌 등) | ❌ | 환경 문제로 오인되어 디버깅 불가 |
| 기동은 되나 동작 이상 | ✅ | 권장 형태 |

### 4-2. 재현 안정성

- 버그 재현이 **머신 사양에 의존하면 안 됨**
  - 타이밍 의존 버그(ZK session timeout 등)는 `TICK_TIME` 조정으로 보정
  - "고사양 머신에서는 안 터짐"이 되면 채점 불가
- OOM 버그는 heap을 충분히 작게 설정해 **어느 환경에서도 재현**되도록 설계

### 4-3. 버그 수 및 독립성

- 문제당 버그 **1~2개**
- 버그는 **서로 독립적으로 특정** 가능해야 함
  - 버그 A를 고쳤을 때 버그 B가 드러나는 계단식 구조 허용
  - 버그 A와 B를 동시에 고쳐야만 재현 가능한 구조 금지

## 5. 기술 스택 허용 범위

| 기술 | 허용 여부 | 조건 |
|---|---|---|
| docker-compose | ✅ 항상 허용 | |
| Kafka + Zookeeper | ✅ | 브로커 1개 |
| Spark Standalone | ✅ | worker 1개, memory 1g |
| PostgreSQL | ✅ | |
| Airflow | ✅ | webserver + scheduler만 (worker 분리 금지) |
| k3d / k8s | ⚠️ 조건부 | 강의에서 해당 내용을 다룬 경우만 |
| Spark Operator | ⚠️ 조건부 | 강의에서 다룬 경우만 |
| Databricks | ✅ | 클라우드 리소스이므로 로컬 사양 무관 |

### M1/M2 Mac 이미지 호환성 체크

아래 이미지는 `linux/amd64` 전용이므로 M1/M2에서 `--platform linux/amd64` 옵션 또는
ARM 호환 이미지로 교체 필요:

- `confluentinc/cp-kafka`, `confluentinc/cp-zookeeper` → 7.4+ 버전은 multi-arch 지원
- `provectuslabs/kafka-ui` → multi-arch 지원
- 커스텀 Spark 이미지 → Dockerfile의 베이스 이미지 확인 필요

## 6. 문제 완성 후 검증 체크리스트

문제를 만든 후 아래 항목을 **직접 실행**해서 확인한다.

### 환경 검증

- [ ] `docker compose up -d` 후 전체 컨테이너 기동 완료 (60초 이내)
- [ ] `docker stats` 기준 전체 메모리 사용량 3.5GB 이하
- [ ] `docker ps` 에서 의도치 않은 `Exited` 컨테이너 없음

### 버그 검증

- [ ] 버그 상태에서 의도된 에러 재현 확인
- [ ] 버그 수정 후 정상 동작 확인
- [ ] 버그 재현이 참조_src 코드만으로 가능 (별도 스크립트 불필요)

### 채점 연계 검증

- [ ] 제출물 경로가 채점기준서와 일치
- [ ] 자동채점 항목의 기댓값(row count, checksum 등) 사전 계산 완료
- [ ] `date && hostname` 출력이 필요한 캡처 단계 명확히 안내
