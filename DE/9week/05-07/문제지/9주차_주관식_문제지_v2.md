## 문제 2: 센서 데이터 Producer 작성

### 목표

제공된 Kaggle 센서 데이터셋(`sensor_data.csv`)을 읽어, 각 센서 측정값을 Kafka `sensor-data` 토픽으로 전송하는 Python 프로듀서 스크립트를 작성합니다.

### 요구사항

**`problem-1-4/sensor_producer.py`** 파일을 작성합니다.

1. **Kafka Producer 초기화**
   - `bootstrap_servers`: `kafka:29092`
   - `value_serializer`: `lambda v: json.dumps(v).encode('utf-8')`

2. **CSV 읽기**
   - `pandas.read_csv("sensor_data.csv")`로 데이터셋을 로드합니다.
   - 각 행을 순회하며 **wide format → long format**으로 변환하여 전송합니다.  
     (한 행에서 4개 센서값 → 4개의 메시지 생성)

3. **메시지 스키마** — 각 센서값마다 아래 필드를 포함하는 JSON 메시지를 생성합니다.

   | 필드 | 설명 | 예시 |
   |------|------|------|
   | `machine_id` | `LINE-{본인이름}-{Product ID}` 형식 | `LINE-홍길동-M14860` |
   | `sensor_type` | 센서 종류 (아래 4가지 중 하나) | `air_temperature` |
   | `value` | 센서 측정값 (CSV 원본 그대로) | `305.2` |
   | `timestamp` | 전송 시점 ISO 8601 (`datetime.now()`) | `2025-05-07T14:30:00Z` |

   > ⚠️ **`machine_id`의 `{본인이름}` 부분에 반드시 본인 이름을 포함하세요.**  
   > Kafka UI 캡처에서 본인 이름이 식별 가능해야 합니다.

4. **센서 종류 및 이상 임계값**

   | `sensor_type` | CSV 컬럼 | 단위 | 이상 임계값 |
   |---------------|----------|------|-------------|
   | `air_temperature` | `Air temperature [K]` | K | **> 305 K** |
   | `process_temperature` | `Process temperature [K]` | K | **> 312 K** |
   | `torque` | `Torque [Nm]` | Nm | **> 65 Nm** |
   | `tool_wear` | `Tool wear [min]` | min | **> 200 min** |

5. **메인 실행 루프**
   - CSV의 모든 행을 순회하며 `sensor-data` 토픽으로 전송합니다.
   - `time.sleep(0.1)` 등으로 전송 간격을 조절합니다.
   - 끝까지 도달하면 처음부터 반복(무한 루프) 또는 1회 실행 후 종료 자유 선택
   - `try...finally`로 `producer.flush()` + `producer.close()` 보장

**메시지 예시**

```json
{
  "machine_id": "LINE-홍길동-M14860",
  "sensor_type": "air_temperature",
  "value": 305.2,
  "timestamp": "2025-05-07T14:30:00Z"
}
```

### 제출물

1. `problem-1-4/sensor_producer.py`
2. `problem-1-4/kafka_ui_sensor_data.png`
   - `sensor-data` 토픽 확인 화면 (토픽명, 메시지 수 포함)
   - 메시지 내용에서 **본인 이름(`machine_id`)이 식별 가능**해야 합니다.

## 문제 5: `click_producer.py` 스크립트 작성 - 데이터 생성 및 Kafka 전송 로직

### 목표

가상의 웹사이트 클릭 스트림 데이터를 생성하고, 이를 JSON 형식으로 Kafka의 `click_stream` 토픽으로 실시간 전송하는 Python 프로듀서 스크립트 (`click_producer.py`)의 핵심 로직을 작성합니다.

### 요구사항

`problem-5/click_producer.py` 파일을 새로 작성하거나 제공된 기본 파일을 수정합니다. 스크립트는 다음 기능을 포함해야 합니다.

1. **라이브러리 import**: `kafka-python` (Kafka Producer용), `json` (JSON 직렬화용), `time`, `random`, `faker` (가상 데이터 생성용) 라이브러리를 import 합니다.

2. **Kafka Producer 초기화**
   - `KafkaProducer` 객체를 생성합니다.
   - `bootstrap_servers`를 `docker-compose.yml` 환경 내의 Kafka 내부 접속 주소(예: `'kafka:29092'`)로 설정합니다.
   - `value_serializer=lambda v: json.dumps(v).encode('utf-8')`와 같이 설정합니다.

3. **클릭 이벤트 데이터 생성 함수**

   다음 필드를 포함하는 Python 딕셔너리 형태의 클릭 이벤트를 생성하는 함수(예: `generate_click_event()`)를 정의합니다.

   - `user_id`: `fake.uuid4()` 등을 사용하여 생성된 고유 사용자 ID (문자열)
   - `page_url`: `fake.uri()` 등을 사용하여 생성된 가상 페이지 URL (문자열)
   - `click_time`: 현재 시간 또는 랜덤 시간을 ISO 8601 형식 (예: `YYYY-MM-DDTHH:MM:SSZ`)의 문자열로 표현 (예: `time.strftime('%Y-%m-%dT%H:%M:%SZ', time.gmtime())`)
   - `referrer`: `fake.uri()` 등을 사용하여 생성된 이전 페이지 URL (문자열). 단, 약 30% 확률로는 `None` 또는 빈 문자열이 되도록 하여 현실성을 반영할 수 있습니다.
   - `user_agent`: `fake.user_agent()` 등을 사용하여 생성된 가상 사용자 에이전트 문자열.

4. **메인 실행 루프**

   무한 루프(또는 충분히 긴 시간 동안 실행되는 루프) 내에서 다음을 반복합니다.

   - 위에서 정의한 함수를 호출하여 클릭 이벤트 데이터를 생성합니다.
   - 생성된 데이터를 `producer.send('click_stream', value=메시지)`를 사용하여 Kafka의 `click_stream` 토픽으로 전송합니다.
   - 터미널에 현재 어떤 토픽으로 메시지를 전송하고 있는지 시작 시 한 번 출력합니다.
   - 약 1초마다 해당 초 동안 전송한 메시지 수를 카운트하여 터미널에 로깅합니다. (예: `"Producer: Sent X messages in the last second to click_stream."`) — 정확한 초당 전송률은 다음 문제에서 확인합니다.
   - `time.sleep(random.uniform(0.05, 0.1))` 등을 사용하여 초당 약 10~20개의 메시지가 전송되도록 전송 간격을 조절합니다.

5. **예외 처리**

   - `KafkaProducer` 초기화 시 발생할 수 있는 오류(예: Kafka 서버 접속 불가 - `kafka.errors.NoBrokersAvailable` 등)에 대한 `try-except` 처리 및 적절한 로깅/에러 메시지 출력을 포함해야 합니다.
   - 메시지 전송 (`producer.send()`) 시 발생할 수 있는 오류에 대한 `try-except` 처리 및 로깅을 포함해야 합니다.
   - 메인 루프 등 주요 로직 실행 중 예상치 못한 오류 발생에 대비한 포괄적인 예외 처리(최상위 `try-except Exception as e`)를 포함하여 프로그램이 비정상적으로 중단되는 것을 방지하고, 오류 발생 시 관련 정보를 로깅해야 합니다.

6. **예외 처리 및 리소스 정리**

   - `try...except KeyboardInterrupt...finally` 구문을 사용하여, `Ctrl+C` 등으로 스크립트가 중단될 때뿐만 아니라, 다른 예외 상황 발생 시에도 `producer.flush()` 및 `producer.close()`가 호출되어 Kafka 프로듀서 관련 리소스가 정상적으로 해제되도록 합니다.

## 문제 7: `click_analyzer.py` 작성 - Spark 환경설정, Kafka Source, Top 페이지 분석 로직

### 목표

Kafka의 `click_stream` 토픽에서 실시간으로 데이터를 소비하고, 최근 1분 동안 가장 많이 클릭된 **Top 5 페이지**를 분석하는 Spark Structured Streaming 애플리케이션 (`click_analyzer.py`)의 기본 골격을 작성합니다.

### 요구사항 (작성)

`problem-7/click_analyzer.py` 파일을 새로 작성하거나 제공된 기본 파일을 수정합니다. 스크립트는 다음 기능을 포함해야 합니다.

1. **라이브러리 Import**
   - `pyspark.sql.SparkSession`
   - `pyspark.sql.functions` (예: `from_json`, `col`, `window`, `desc`, `udf`)
   - `pyspark.sql.types` (예: `StructType`, `StructField`, `StringType`, `TimestampType`)

2. **SparkSession 초기화**
   - `SparkSession.builder`를 사용하여 SparkSession 객체를 생성합니다.
   - `appName`을 `"ClickStreamAnalyzer"` (또는 유사한 이름)으로 설정합니다.
   - `master`를 `docker-compose.yml`에 정의된 Spark 마스터 URL (예: `spark://spark-master:7077`)로 설정합니다.
   - Kafka와 Spark 연동을 위한 `jars.packages`를 설정합니다 (예: `org.apache.spark:spark-sql-kafka-0-10_2.12:3.3.0` — Spark 버전에 맞는 패키지 사용).

3. **Kafka Source 설정 및 데이터 스키마 정의**
   - `spark.readStream`을 사용하여 Kafka Source를 정의합니다.
   - `kafka.bootstrap.servers`를 Kafka 내부 접속 주소 (예: `kafka:29092`)로 설정합니다.
   - `subscribe` 옵션으로 `click_stream` 토픽을 지정합니다.
   - `startingOffsets`를 `latest`로 설정하여 최신 데이터부터 처리하도록 합니다.
   - 클릭 이벤트 데이터의 스키마(`StructType`)를 정의합니다. `click_producer.py`에서 생성한 필드(`user_id`, `page_url`, `click_time`, `referrer`, `user_agent`)와 데이터 타입을 고려해야 합니다. 특히 `click_time`은 문자열로 수신됩니다.

4. **데이터 파싱 및 Timestamp 변환**
   - Kafka에서 가져온 `value` (binary)를 문자열로 변환한 후, `from_json` 함수와 정의된 스키마를 사용하여 JSON 데이터를 파싱합니다.
   - 파싱된 데이터에서 `click_time` (문자열) 필드를 `TimestampType`으로 변환하여 새로운 컬럼(예: `event_timestamp`)을 추가합니다. 필요시 `to_timestamp` 함수를 사용합니다.

5. **Top 5 페이지 분석 로직** (최근 1분, 5초마다 업데이트)
   - `event_timestamp`를 기준으로 1분 길이의 슬라이딩 윈도우(`windowDuration="1 minute"`, `slideDuration="5 seconds"`)를 적용합니다.
   - 윈도우별로 `page_url`을 그룹화하고, 각 페이지의 클릭 수를 계산합니다 (`count`).
   - 클릭 수를 기준으로 내림차순 정렬 (`desc("count")`) 합니다.
   - 상위 5개 결과만 선택합니다 (`limit(5)`).

6. **결과 출력 (콘솔)**
   - 분석 결과를 콘솔에 출력하기 위해 `writeStream`을 설정합니다.
   - `outputMode`를 `"complete"` (또는 상황에 맞는 다른 모드)로 설정합니다.
   - `format`을 `"console"`로 설정합니다.
   - `truncate=False` 옵션을 사용하여 긴 URL이 잘리지 않도록 합니다.
   - 스트리밍 쿼리가 시작된 후, `query.awaitTermination()`을 호출하여 스트리밍 쿼리가 종료될 때까지 대기합니다.

7. **예외 처리**
   - `SparkSession` 초기화, Kafka Source 설정 등 주요 설정 단계에서 발생할 수 있는 오류에 대해 `try-except` 블록을 사용하여 처리하고, 오류 발생 시 적절한 로그와 함께 프로그램을 정상적으로 종료하도록 합니다.
   - 데이터 파싱(예: 잘못된 JSON 형식, 스키마 불일치) 또는 Timestamp 변환 과정에서 발생할 수 있는 오류를 처리하기 위해, 해당 로직 부분에 `try-except`를 사용하거나 Spark의 오류 처리 기능(예: `badRecordsPath` 옵션, `FAILFAST` 모드 대신 `PERMISSIVE` 모드 사용 후 null 값 처리 등)을 활용하여 스트리밍 작업 전체가 중단되지 않도록 견고하게 작성해야 합니다. (이 문제에서는 `PERMISSIVE` 모드 사용 및 null 값 필터링/별도 처리 방식을 권장합니다.)
   - 메인 스크립트 실행 부분을 `try...finally`로 감싸 SparkSession이 비정상 종료 시에도 `spark.stop()`이 호출되도록 보장합니다.

   - 터미널에 Top 3 사용자의 `user_id`와 해당 사용자의 `count` (클릭 수)가 포함된 분석 결과가 최소 2~3 주기 이상 (5초 간격으로 업데이트되는 내용 포함) 명확하게 보여야 합니다.
   - (선택 사항) 만약 Top 페이지 분석 결과도 함께 출력되도록 구현했다면, 해당 부분도 스크린샷에 포함되어도 괜찮습니다.

### 참고

- 두 가지 분석(Top 페이지, Top 사용자)을 하나의 `click_analyzer.py`에서 동시에 수행하고 각각의 결과를 별도의 콘솔 스트림으로 출력하는 것이 일반적입니다. 각 `writeStream` 쿼리에 대해 다른 `queryName`을 지정하면 관리에 용이합니다.
- 기존 Top 페이지 분석 로직을 참고하여 Top 사용자 분석 로직을 유사하게 구현할 수 있습니다. 그룹화 대상 컬럼과 limit 값만 달라집니다.
- 여러 스트리밍 쿼리를 동시에 실행하고 관리하기 위해 `spark.streams.awaitAnyTermination()`을 사용하는 것이 좋습니다. 또는 각 쿼리 객체에 대해 `awaitTermination()`을 순차적으로 호출하면, 첫 번째 쿼리가 종료되어야 다음 쿼리가 시작되므로 동시 실행을 테스트하기 어렵습니다. 이 경우, 테스트 시에는 하나의 쿼리만 `awaitTermination()`을 호출하고 다른 쿼리는 `start()`만 하여 백그라운드로 실행되도록 할 수 있습니다.

## 문제 9: `click_analyzer.py` 확장 - 의심스러운 활동 패턴 감지 및 Kafka 전송 로직 추가

### 목표

`click_analyzer.py` 스크립트에 새로운 분석 로직을 추가하여, 특정 사용자 행동 패턴(예: **30초 이내에 동일 사용자가 5회 이상 서로 다른 페이지를 클릭**하는 경우)을 '의심스러운 활동'으로 감지하고, 그 결과를 새로운 Kafka 토픽(`suspicious_activity`)으로 전송하는 기능을 구현합니다.

### 요구사항 (작성)

- **사전 조건**: 앞 문제들의 모든 요구사항이 성공적으로 완료되어 있어야 합니다.

- `problem-9/click_analyzer.py` 파일을 수정합니다. 기존 분석 로직(Top 페이지, Top 사용자)은 유지하거나 주석 처리하고, 새로운 의심 활동 감지 및 Kafka 전송 로직을 추가합니다.

1. **의심스러운 활동 패턴 감지 로직** (30초 윈도우, 10초마다 업데이트 또는 각 배치마다 처리)
   - 파싱된 데이터(`parsed_df` 또는 `selected_df`)를 사용합니다.
   - `event_timestamp`를 기준으로 30초 길이의 슬라이딩 윈도우(`windowDuration="30 seconds"`, `slideDuration="10 seconds"` 또는 각 배치마다 처리하는 방식)를 적용합니다.
   - 윈도우별로 `user_id`를 그룹화하고, 각 사용자가 클릭한 고유한 페이지 수(`page_url`)를 계산합니다 (`countDistinct("page_url")` 사용).
   - 계산된 고유 페이지 수가 5 이상인 경우(`>= 5`)를 '의심스러운 활동'으로 간주하고 필터링합니다.
   - 필터링된 결과에는 `user_id`, `window`(시작/종료 시간), 감지된 `distinct_page_count`가 포함되도록 합니다.

2. **감지 결과 Kafka 토픽(`suspicious_activity`)으로 전송**
   - 위에서 감지된 의심스러운 활동 결과를 `suspicious_activity`라는 새로운 Kafka 토픽으로 전송합니다.
   - 이를 위해 `foreachBatch` 연산을 사용하고, 그 안에서 Python 함수를 정의하여 Kafka Producer를 통해 메시지를 전송합니다.
   - Python 함수 내에서 `KafkaProducer`를 초기화합니다 (bootstrap servers, value serializer 설정 — JSON 형식 권장).
   - 처리할 배치 DataFrame(`batch_df`)이 비어 있지 않은 경우, `batch_df.collect()`를 통해 결과를 가져와 각 row를 Kafka 메시지로 전송합니다.
   - 메시지에는 `user_id`, `window_start`, `window_end`, `distinct_page_count` 등의 정보가 포함되도록 합니다.
   - Producer 사용 후에는 반드시 `close()`를 호출하여 리소스를 정리합니다.
   - 터미널에는 의심스러운 활동이 감지되어 Kafka로 전송될 때 관련 정보를 로깅하도록 합니다 (예: 감지된 사용자 ID, 전송 내용).

3. **결과 출력 (콘솔 — 선택 사항)**
   - 의심 활동 감지 결과를 콘솔에도 간략히 출력하는 `writeStream` 쿼리를 추가할 수 있으나, Kafka 전송이 주요 목표입니다. 만약 콘솔 출력을 추가한다면, `outputMode`는 `"update"` 또는 `"append"`가 적절할 수 있습니다.

### 요구사항 (실행 및 결과 확인)

수정된 `click_analyzer.py` 스크립트를 실행하고, 의도적으로 의심스러운 활동 패턴을 발생시키면서 터미널 로그와 Kafka UI를 통해 패턴 감지 및 `suspicious_activity` 토픽으로의 결과 전송을 확인합니다.

- **사전 조건**: 앞 문제들의 모든 요구사항이 성공적으로 완료되어 있어야 합니다. (Docker 환경 실행 중, `click_producer.py` 및 수정된 `click_analyzer.py` 스크립트 준비 완료)

- **의심스러운 활동 패턴 유도** (방법 선택)
   - **방법 1**: `click_producer.py` 일시적 수정  
     `click_producer.py` 스크립트를 잠시 수정하여, 특정 `user_id` (예: `suspicious_user_123`)가 짧은 시간(예: 10~20초) 동안 매우 다양한 `page_url` (최소 5개 이상)을 클릭하도록 데이터를 생성합니다. `time.sleep()` 간격을 줄이고, `page_url`을 매번 다르게 생성하도록 로직을 변경할 수 있습니다. 수정된 `click_producer.py`를 실행합니다.
   - **방법 2**: 별도의 간단한 프로듀서 스크립트 작성 및 실행  
     의심스러운 활동 데이터만 `click_stream` 토픽으로 전송하는 간단한 Python 스크립트를 작성하여 실행합니다. 이 스크립트는 특정 `user_id`에 대해 짧은 시간 내에 여러 다른 `page_url`을 가진 메시지를 전송해야 합니다.
   - **방법 3**: 수동 데이터 주입 (어려울 수 있음)

- **수정된 `click_analyzer.py` 실행**: `click_analyzer.py` (의심 활동 감지 로직이 추가된 버전)를 `spark-submit` 명령어를 사용하여 실행합니다.

   - **Kafka UI 확인**: Kafka UI에 접속하여 `suspicious_activity` 토픽이 생성되었는지 확인합니다. 해당 토픽을 선택하고, 메시지가 정상적으로 적재되었는지 확인합니다. 적재된 메시지의 내용을 확인하여 `user_id`, 시간 정보, `distinct_page_count` 등이 `click_analyzer.py`의 터미널 로그와 일치하는지, 그리고 의도한 패턴과 부합하는지 확인합니다.

### 참고

- `foreachBatch`는 Structured Streaming에서 각 마이크로 배치의 출력을 외부 시스템(여기서는 Kafka)으로 전송할 때 유용하게 사용됩니다.
- `foreachBatch` 내에서 Kafka Producer를 생성하고 사용하는 경우, Producer 인스턴스가 각 배치마다 새로 생성되고 닫히도록 주의해야 리소스 누수를 방지할 수 있습니다. (함수 내에서 생성 및 close)
- 의심스러운 활동 패턴을 발생시키기 위해 `click_producer.py`를 잠시 수정하거나, 수동으로 특정 사용자가 짧은 시간 내에 여러 다른 페이지를 클릭하는 것처럼 데이터를 생성하여 테스트해볼 수 있습니다.
- `countDistinct` 함수를 사용하기 위해 `pyspark.sql.functions`에서 import 해야 합니다.
- 의심스러운 활동 패턴을 정확히 유도하는 것이 이 문제의 핵심 중 하나입니다. `click_producer.py`를 수정하는 경우, 다른 사용자의 일반적인 클릭 데이터도 계속 생성되도록 하면서 특정 사용자의 행동만 변경하는 것이 좋습니다.
- `click_analyzer.py`의 윈도우 크기(30초)와 필터 조건(고유 페이지 5개 이상)을 고려하여 패턴을 유도해야 합니다.
- Kafka UI에서 메시지 내용을 확인할 때, JSON 형식으로 전송했다면 UI에서 해당 메시지를 쉽게 읽을 수 있습니다.
- 만약 패턴 유도가 어렵거나 결과가 명확하지 않으면, `click_analyzer.py`의 로직(윈도우 크기, 필터 조건, Kafka 전송 로직)을 다시 한번 점검하고 디버깅해야 합니다.
